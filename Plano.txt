Helena Semantic Studio — Implementation Plan Revisado

Aplicação fullstack para gerenciar, testar e validar a primeira camada semântica da IA Helena.

O projeto será uma base técnica funcional e extensível para administrar domínios, métricas, skills e simulações semânticas, gerando um semantic plan estruturado que futuramente poderá ser enviado aos workflows da IA Helena.

==================================================
1. Stack
==================================================

Backend:
- Node.js
- Fastify
- TypeScript
- Prisma ORM
- PostgreSQL
- Zod
- @fastify/cors

Frontend:

- Interface em português brasileiro

Infra:

Observação sobre Tailwind/shadcn:
Usar uma versão do Tailwind CSS compatível com shadcn/ui. Caso Tailwind v4 gere incompatibilidade, usar a configuração estável recomendada pelo shadcn/ui.

==================================================
2. Objetivo do Projeto
==================================================

Criar o Helena Semantic Studio, uma aplicação administrativa para:

- Cadastrar domínios semânticos.
- Cadastrar métricas semânticas.
- Cadastrar skills.
- Simular perguntas em linguagem natural.
- Gerar semantic plans estruturados.
- Salvar histórico de testes semânticos.
- Exibir dashboard com estatísticas básicas.
- Preparar a arquitetura para integração futura com IA Helena.

A primeira versão não deve integrar com:
- Power BI real.
- LLM real.
- Workflows reais da Helena.
- Senior.
- Supabase.
- n8n.
- APIs internas reais.

A aplicação deve deixar a base preparada para isso futuramente.

14. Semantic Plan Logic
==================================================

A lógica inicial do semantic plan deve funcionar sem LLM.

Fluxo:

1. Receber input.
2. Normalizar texto:
   - lowercase
   - remover acentos
   - trim
   - remover espaços duplicados
3. Buscar métricas ativas no banco.
4. Para cada métrica:
   - comparar nome normalizado.
   - comparar synonyms normalizados.
   - considerar domínio relacionado.
5. Selecionar métrica com maior score.
6. Detectar período:
   - hoje → { raw: "hoje", type: "relative", value: "today" }
   - ontem → { raw: "ontem", type: "relative", value: "yesterday" }
   - mês atual / mes atual → { raw: "mês atual", type: "relative", value: "current_month" }
   - semana → { raw: "semana", type: "relative", value: "current_week" }
   - safra → { raw: "safra", type: "business_period", value: "harvest" }
7. Detectar agrupamentos:
   - turno → shift
   - dia → day
   - mês / mes → month
   - fazenda → farm
   - frente → front
   - equipamento → equipment
8. Sugerir skills:
   - Sempre sugerir metric-resolver quando encontrar métrica.
   - Sugerir ranges-relativos quando encontrar período.
   - Sugerir turnos-rules quando groupBy incluir shift.
   - Sugerir chart-recommendation quando input tiver gráfico, grafico, visualizar, tendência, tendencia.
   - Sugerir data-freshness quando input tiver atualizado, atualização, atualizacao, defasagem.
   - Sugerir answer-verifier sempre que encontrar métrica.
9. Calcular confidence com regra simples:
   - Sinônimo exato encontrado: +0.50
   - Nome da métrica encontrado: +0.30
   - Período detectado: +0.10
   - Agrupamento detectado: +0.10
   - Domínio detectado: +0.10
   - Limitar máximo em 1.0
10. Se nenhuma métrica for encontrada ou confidence < 0.5:
   - needsClarification = true
   - metric = null
   - domain = null se não houver domínio detectado
11. Salvar SemanticTest:
   - input
   - generatedPlan
   - confidence
   - needsClarification
12. Retornar o plano.

- O backend inicia sem erro.
- O frontend inicia sem erro.
- O Prisma conecta no banco.
- As migrations funcionam.
- O seed funciona.
- GET /health retorna sucesso.
- GET /api/dashboard/stats retorna totais.
- GET /api/domains retorna domínios.
- GET /api/metrics retorna métricas com domínio.
- GET /api/skills retorna skills.
- POST /api/semantic/plan gera plano semântico.
- O teste semântico é salvo no banco.
- GET /api/semantic/history retorna histórico.
- O frontend consome o backend.
- A página /dashboard exibe os totais reais.
- A página /semantic gera plano semântico.
- A página /history exibe testes salvos.
- A estrutura está modular e pronta para crescer.

- O README permite que outra pessoa rode o projeto do zero.

- Não há integração externa real.
- Não há dados sensíveis.

18. Observações Finais
==================================================

Este projeto deve ser implementado como uma base real e funcional para o Helena Semantic Studio.

A aplicação deve ser simples, limpa e extensível.

O foco não é criar toda a IA Helena agora.

O foco é criar a primeira interface administrativa e técnica da camada semântica, com:

- domínios
- métricas
- skills
- simulação de pergunta
- semantic plan
- histórico
- dashboard

Futuramente, essa aplicação poderá evoluir para:

- consumir catálogos reais da Helena
- importar skills reais
- sugerir workflows
- chamar LLM
- conectar Power BI
- conectar n8n
- conectar APIs internas
- executar semantic plans em produção
